### Q24 — DT(Dependency-Track) 연동 전략

**v1 문제의 근본 원인**: DT는 Java/Spring 앱으로 4GB+ RAM을 요구하고, 재시작 시 포털의 polling이 타임아웃되며, 포털 DB와 DT DB 간 상태 불일치가 발생합니다.

**v2 권장 방향: "DT 유지 + 안정화 레이어 추가"**

```
[Portal]
  ├── DT Health Monitor (60초 간격 heartbeat)
  │     └── unhealthy 감지 시 → docker restart + 알림
  ├── Circuit Breaker (DT API 호출 전 상태 확인)
  │     └── OPEN 상태 시 → PostgreSQL 캐시 데이터 제공
  ├── Vulnerability Cache (PostgreSQL)
  │     └── DT 응답을 캐싱 → DT 재시작 중에도 포털 정상 동작
  ├── Orphan Cleaner (6시간 스케줄)
  │     └── DT 고아 프로젝트 자동 감지 + Admin UI 표시
  └── Sync Status Dashboard (Admin)
        └── DT 연결 상태, 마지막 동기화 시각, 불일치 카운트
```

DT를 교체하지 않는 이유: NVD/OSV/GitHub Advisory 통합이 완성도 높고, 재구현 비용이 3개월+이며, DT의 취약점 분석 품질이 검증됨.

---

### Q25 — 팀/조직 모델 제안

Black Duck / Snyk / Mend 패턴 참조:

```
Organization (1 per deployment)
├── Super Admin  ← 시스템 전체 관리 (배포자/IT관리자)
│
├── Team A
│   ├── Team Admin  ← 팀원 추가/제거, 팀 설정
│   ├── Developer   ← 프로젝트 스캔·조회
│   └── Projects: [앱A, 앱B, 앱C]
│
└── Team B
    ├── Team Admin
    ├── Developer (read-only)
    └── Projects: [앱D]
```

| 역할 | 범위 | 권한 |
|------|------|------|
| Super Admin | 전체 | 사용자·팀·시스템·DT 설정 모두 |
| Team Admin | 소속 팀 | 팀원 관리, 프로젝트 생성·삭제 |
| Developer | 소속 팀 프로젝트 | 스캔 실행, 결과 조회, SBOM 다운로드 |

프로젝트 가시성: `team-only`(기본) / `org-wide`(설정 시 전체 팀 조회 가능)
데모 SaaS: 가입 시 개인 Team 자동 생성 → 본인 프로젝트만 접근, Super Admin은 전체 조회.


## 2. 스캔 방식 충분성 분석

### v1 스캔 파이프라인

```
Git Repo
  → cdxgen       SBOM 생성 (의존성 탐지)
  → ORT          라이선스 분석·평가
  → DT           취약점 모니터링 (NVD/OSV)
```

### 강점과 한계

| 항목 | 평가 | 상세 |
|------|------|------|
| 의존성 탐지 정확도 | ⚠ 중상 | cdxgen은 npm/Maven/pip는 우수, 일부 언어는 낮음 |
| 라이선스 분석 | ✅ 우수 | ORT는 업계 최고 수준 |
| 취약점 데이터 | ✅ 양호 | NVD + OSV + GitHub Advisory |
| 스캔 속도 | ⚠ 느림 | ORT source 모드 10~60분 |
| 컨테이너 커버리지 | ❌ 없음 | OS 패키지 미탐지 |
| 소스 코드 조각 탐지 | ❌ 없음 | copy-paste OSS 미탐지 |
| 직접 vs 전이 의존성 | ✅ 지원 | ORT가 구분 |

### 제안: v2 스캔 파이프라인
