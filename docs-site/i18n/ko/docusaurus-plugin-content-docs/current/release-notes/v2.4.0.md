# v2.4.0 — Trivy 단일 엔진 + 엔터프라이즈 폴리시

**릴리스일:** 2026-06-XX

GA 이후 가장 큰 아키텍처 릴리스입니다. Dependency-Track을 **제거**하고 Trivy를 단일 임베디드 취약점 엔진으로 채택했으며, UI는 3개월간의 디자인·IA 작업(W9–W11)을 통합해 모던 엔터프라이즈 미학 — 전용 대시보드, drawer + page-nav dual surface, ⌘K 커맨드 팔레트, filter / columns picker, Vercel-base + Linear-polish 톤 — 으로 정돈했습니다.

:::warning Breaking changes
v2.4.0은 Dependency-Track을 제거합니다. v2.3.x에서 업그레이드하는 운영자는 `bash scripts/upgrade.sh` 실행 **전** [v2.3.x에서의 마이그레이션](#migration-from-v23x)을 반드시 읽어야 합니다. v2.3.1은 SECURITY.md backport 정책으로 계속 지원됩니다.
:::

## 하이라이트

### Trivy가 단일 취약점 엔진

`v2.4.0`은 Dependency-Track을 제거하고 SBOM → CVE 매칭에 Trivy를 직접 사용합니다. 주간 Celery beat이 Trivy DB를 갱신하고, 자동 재매칭 태스크가 DB 도착 시 모든 프로젝트의 최신 SBOM을 순회 — 사용자는 재스캔 없이 새 finding을 확인합니다.

- **메모리 풋프린트 −50 %** — 4 GB JVM 제거. 스택이 4 GB 호스트에 여유 있게 들어맞음.
- **데이터 출처 변동 없음** — NVD + OSV + GHSA + EPSS + KEV. DT 대신 Trivy DB로 전달.
- **자동 재매칭 beat** — DT가 우리 배포에서 제공 못 한 기능. 네이티브로 탑재.
- **Air-gapped 1급 지원** — `TRIVY_DB_REPOSITORY`로 업스트림 OCI 레지스트리를 사내 미러로 교체.

운영은 [취약점 데이터 (Trivy DB)](../admin-guide/vulnerability-data.md), 출처별 매트릭스는 [데이터 출처](../reference/data-sources.md), 결정 배경은 [ADR-0001](https://github.com/trustedoss/trustedoss-portal/blob/main/docs/decisions/0001-replace-dt-with-trivy.md) 참조.

루트 URL `/`는 이제 프로젝트 목록 대신 전용 **Dashboard**를 노출합니다 — 전사 리스크 포트폴리오, severity 분포, 최근 스캔, at-a-glance KPI.

Vercel-base + Linear-polish — 색상 램프 리프레시, 더 타이트한 밀도, hover / focus / motion 마이크로인터랙션, 맞춤형 빈 상태 일러스트. 아래 Vulnerabilities 탭은 v2.4.0 어포던스(Add filter, Columns picker, 탭 헤더에 고정된 distribution 카드)와 새 톤을 함께 보여줍니다:

### Drawer + page-nav dual surface (W10)

취약점 상세는 두 경로로 도달 가능합니다 — 빠른 트리아지용 slide-over drawer, 그리고 deep-link / 뒤로가기 안전 내비게이션을 위한 전용 페이지 라우트. 둘은 동일 콘텐츠를 공유하며, 페이지 라우트는 "References" 섹션과 **Next steps** 액션 패턴도 복원합니다.

### Filter, columns, 팔레트 (W9-#52 / #54)

모든 목록 탭(Vulnerabilities, Components, Projects, Scans)에 세 가지가 추가됩니다:

- **+ Add filter** chip — 발견 가능, 다중 필드, URL에 영속.
- **Columns picker** — 목록 컬럼 표시 / 숨김 / 재정렬. 사용자별 저장.
- **⌘K 커맨드 팔레트** — 모든 페이지에서 프로젝트, 스캔, 설정, admin 영역 검색.

### Maven classifier 인식 스캔 (W8-#46)

`classifier`(`-sources`, `-javadoc`, `-tests`, 커스텀 shadow JAR)가 있는 Maven 아티팩트가 이제 올바르게 스캔됩니다. 이전 빌드는 classifier 변형을 base 좌표로 합쳤지만, v2.4.0은 `purl`에서 classifier를 보존해 변형별로 CVE를 매칭합니다.

## Breaking changes

### Dependency-Track 제거

DT 컨테이너, `/admin/dt` 페이지, `dt_*.py` Celery 태스크, `integrations/dt/` Python 모듈, `docker-compose.dt.yml` overlay가 모두 삭제됩니다. CVE 매칭은 이제 `apps/backend/integrations/trivy.py` → `trivy sbom` → `vulnerability_findings`로 흐릅니다.

의미:

- **기존 데이터 보존.** v2.3의 `vulnerability_findings` 행은 그대로 유지. 업그레이드 후 자동 재매칭 beat이 다음 갱신 시 Trivy로 재채움 — 보통 수 시간 내.
- **프론트엔드 `/admin/dt` 제거.** 운영자는 **/admin/health**로 착지. 곧 도착할 **Vulnerability data** 카드(W6-#43e 후속)가 Trivy DB 신선도를 여기에 노출.
- **감사 로그 보존.** 과거 `dt_*` 액션 행은 역사 사실로 유지. admin 필터 UI는 deprecated로 표시.

### 환경 변수

| 제거 (DT) | 교체 (Trivy) |
|---|---|
| `DT_URL` | (없음 — Trivy DB는 워커 로컬) |
| `DT_API_KEY` | (없음) |
| `DT_REQUEST_TIMEOUT_SECONDS` | `TRIVY_TIMEOUT_SECONDS` (기본 300) |
| `DT_BREAKER_FAILURE_THRESHOLD` | (없음 — 원격 엔진 없음) |
| `DT_BREAKER_COOLDOWN_SECONDS` | (없음) |
| `DT_HEALTH_ENDPOINT` | (없음) |
| `DT_AUTO_RESTART` | (없음) |
| `DT_ORPHAN_AUTODELETE` | (없음 — 고아 모델 없음) |
| — | `TRIVY_DB_REPOSITORY` (기본 `ghcr.io/aquasecurity/trivy-db`) |
| — | `TRIVY_DB_REFRESH_HOURS` (기본 `168`) |
| — | `TRIVY_DB_CACHE_DIR` (기본 `/var/lib/trivy`) |

`bash scripts/upgrade.sh`는 기존 `.env`의 `DT_*` 키를 자동으로 주석 처리하고, 워커는 `docker-compose.yml`의 `TRIVY_*` 기본값을 읽습니다 — 스크립트 사용 시 수동 편집 불필요. 커스텀 CI / Helm 등 손으로 업그레이드하는 경우는 아래 "수동 `.env` 마이그레이션" 안내 참조.

canonical reference는 [환경 변수 — 취약점 데이터](../reference/env-variables.md#취약점-데이터).

### Helm chart 0.3.0

차트는 **0.3.0**으로 올라가며 `appVersion: "2.4.0"`을 채택합니다. 구체적으로:

- **제거:** `env.dt.url`, `env.dt.apiKey`, 차트 Secret의 `DT_API_KEY`, ConfigMap의 `DT_URL`. (차트는 원래 `dtrack-api` Deployment를 번들한 적이 없으므로 — DT는 항상 외부 — 삭제되는 워크로드 객체는 없습니다.)
- **추가:** `env.trivy.dbRepository`(기본 `ghcr.io/aquasecurity/trivy-db`), `env.trivy.dbRefreshHours`(`168`), `env.trivy.dbCacheDir`(`/var/lib/trivy`), `env.trivy.timeoutSeconds`(`300`).
- **승격:** `image.tag` 기본값 `2.0.0` → `2.4.0`.

In-place 업그레이드:

```bash
helm upgrade trustedoss oci://ghcr.io/trustedoss/charts/trustedoss \
  --version 0.3.0 \
  -n trustedoss \
  -f my-values.yaml
```

기존 `my-values.yaml`에 `env.dt.*` 키가 있다면 제거하세요 — Helm 0.3.0은 그 키를 정의하지 않으므로 strict-mode 검증이 실패합니다. [Helm/Kubernetes 설치](../installation/helm.md) 참조.

### CI 통합 — `scan-kind` 텍스트만 변경

GitHub Action `scan-kind=source` 입력의 `cdxgen + scancode + DT` 설명이 `cdxgen + scancode + Trivy`로 바뀝니다. 계약은 동일 — 기존 워크플로는 수정 없이 계속 동작합니다.

## v2.3.x에서의 마이그레이션 {#migration-from-v23x}

업그레이드의 형태는 **단방향, 수동 데이터 마이그레이션 없음**입니다. 마이그레이션된 워커가 첫 부팅 시 Trivy DB를 재pull하고, beat 태스크가 모든 프로젝트의 최신 SBOM을 재매칭하며, 새 finding이 이후 수 시간 동안 흘러 들어옵니다.

High-level (1회, 호스트에서 실행):

1. **백업.** 업그레이드 전 새 백업 — `/admin/backup → Run manual backup now`, 또는 `bash scripts/backup.sh`. `manifest.json`이 off-host 스토리지에 안착했는지 확인.
2. **v2.4.0 이미지 pull.** `git pull && git checkout v2.4.0`, 이후 `docker-compose pull` (no-clone 설치 경로는 `.env`에서 `IMAGE_TAG=2.4.0` 설정).
3. **`bash scripts/upgrade.sh` 실행.** 스크립트는 v2.3 배포(`.env`에 DT 키 또는 `dtrack-api` 컨테이너 존재)를 자동 감지하고 새 이미지 pull 전 5-step DT 제거 prelude를 수행:

   | Step | 동작 |
   |---|---|
   | 2.1 | `celery inspect active`를 **최대 10분** 폴링하며 실행 중 스캔의 drain을 대기. `UPGRADE_SKIP_DRAIN=1`로 스킵 가능; 타임아웃 후에도 큐가 비지 않으면 prompt 폴백. |
   | 2.2 | `docker rm -f dtrack-api` — 컨테이너 중지+제거; **docker 볼륨은 보존** (2.3에서 archive). |
   | 2.3 | 감지된 DT 볼륨(`*dt-data*`)을 `./backup/dt-volume-<timestamp>.tar.gz`로 archive 할지 prompt. **기본: N** (archive 안 함, 볼륨 그대로 보존). 볼륨은 자동 삭제되지 않습니다. |
   | 2.4 | `.env` 와 `.env.example` 의 모든 `DT_*` 라인을 `# removed in v2.4.0 (W6-#43d)` 마커와 함께 idempotent 주석 처리. |
   | 2.5 | **/admin/health → Trigger full re-match** (W6-#43e)로 즉시 전체 재매칭을 트리거하라는 reminder 출력. 그러지 않아도 6시간 간격 재매칭 beat이 자동 catchup. |

   prelude 후 스크립트는 잔여 `trustedoss.dt_*` 태스크명을 broker에서 purge하고(NACK-loop 가드), v2.4.0 up, `alembic upgrade head`, `/health` 200 대기.

4. **확인.** `/admin/health` 열기 — 모든 행 green. 아무 프로젝트나 열기 — 첫 재매칭 사이클 후 수 분 내 Vulnerabilities 탭 채워짐.
5. **(옵션) DT 볼륨 삭제.** 2.3에서 archive를 받고 누락된 데이터가 없음을 확인했다면, `docker volume rm trustedoss_dt-data` (실제 이름은 `docker volume ls | grep dt`로 확인).

:::info 수동 `.env` 마이그레이션 (커스텀 CI / Helm — `upgrade.sh` 미사용)
위 5-step prelude는 스크립트 없이도 그대로 runbook입니다. `scripts/upgrade.sh` 없이 커스텀 CI / Helm-only 배포라면 2.1 → 2.4를 손으로:

```bash
# 2.1 — drain (best effort; 또는 UPGRADE_SKIP_DRAIN=1)
docker-compose -f docker-compose.yml exec -T worker \
  celery -A tasks.celery_app inspect active

# 2.2 — dtrack-api 컨테이너 중지+제거 (볼륨 보존)
docker rm -f dtrack-api

# 2.3 — DT 볼륨 archive (옵션)
docker run --rm -v trustedoss_dt-data:/data:ro -v "$PWD/backup":/backup \
  busybox:1.36 tar -C /data -czf "/backup/dt-volume-$(date +%Y%m%d).tar.gz" .

# 2.4 — DT_* 키 주석
sed -i.bak -E 's/^(DT_[A-Z_]+=.*)$/# \1   # v2.4.0에서 제거/' .env

docker-compose -f docker-compose.yml up -d
```

이후 `docker-compose logs --tail=200 worker | grep -i trivy`로 확인 — 첫 DB 다운로드는 1~3분 내 안착.
:::

### v2.3.x로 롤백

v2.4.0은 v2.3 역방향 마이그레이션을 제공하지 않습니다. 롤백이 필요하면:

1. 1단계에서 받은 v2.3 백업 복원.
2. `git checkout v2.3.1 && docker-compose pull && docker-compose -f docker-compose.yml -f docker-compose.dt.yml up -d`.
3. DT가 올라올 때까지 대기 (5~10분). 다음 시간별 DT 재싱크가 캐시를 재구축.

`v2.3.1` 릴리스 라인은 SECURITY.md backport 정책에 따라 v2.4.0 GA 이후 6개월간 critical-severity 보안 수정을 받습니다.
