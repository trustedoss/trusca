# v2.4.0 — Trivy single-engine + enterprise polish

**Release date:** 2026-06-XX

The biggest architectural release since GA. Dependency-Track is **removed** in favour of Trivy as the single embedded vulnerability engine, and the UI absorbs three months of design and IA work (W9–W11) into a modern enterprise aesthetic — a dedicated dashboard, drawer + page-nav dual-surface, ⌘K command palette, filter / columns picker, and a polished Vercel-base + Linear-polish look.

:::warning Breaking changes
v2.4.0 removes Dependency-Track. Operators upgrading from v2.3.x **must** read [Migration from v2.3.x](#migration-from-v23x) before running `bash scripts/upgrade.sh`. v2.3.1 remains supported via the SECURITY.md backport policy.
:::

## Highlights

### Trivy is now the single vulnerability engine

`v2.4.0` removes Dependency-Track and uses Trivy directly for SBOM → CVE matching. A weekly Celery beat refreshes the Trivy DB; an automatic re-match task walks every project's most-recent SBOM when the DB lands, so users see fresh findings without re-triggering a scan.

- **Memory footprint −50 %** — 4 GB JVM gone; the stack now fits comfortably on a 4 GB host.
- **Data sources unchanged** — NVD + OSV + GHSA + EPSS + KEV, just delivered through the Trivy DB instead of DT.
- **Automatic re-match beat** — DT could not do this in our deployment; it now ships native.
- **Air-gapped first-class** — `TRIVY_DB_REPOSITORY` swaps the upstream OCI registry for an internal mirror.

See [Vulnerability data (Trivy DB)](../admin-guide/vulnerability-data.md) for operation, [Data sources](../reference/data-sources.md) for the per-source matrix, and [ADR-0001](https://github.com/trustedoss/trustedoss-portal/blob/main/docs/decisions/0001-replace-dt-with-trivy.md) for the decision.

### Dedicated Dashboard at `/` (W9-#50)

The root URL `/` now serves a dedicated **Dashboard** instead of the project list — org-wide risk portfolio, severity distribution, recent scans, and at-a-glance KPIs.

![Dashboard — org-wide severity distribution + recent scans + risk KPIs at the root URL after sign-in](/img/screenshots/release-v2-4-0-dashboard.png)

### Modern enterprise aesthetic (W11)

Vercel-base with Linear-polish — refreshed colour ramps, tighter density, hover / focus / motion microinteractions, and bespoke empty-state illustrations. The Vulnerabilities tab below shows the new tone alongside the v2.4.0 affordances (Add filter, Columns picker, sticky severity distribution card):

![Project detail — Vulnerabilities tab with new color tokens, +Add filter chip, columns picker, and the distribution card pinned to the tab header](/img/screenshots/release-v2-4-0-vulnerabilities.png)

### Drawer + page-nav dual surface (W10)

Vulnerability detail is reachable two ways: the slide-over drawer for fast triage, and a dedicated page route for deep-link and back-button-safe navigation. Both share the same content; the page route also restores the "References" section and the **Next steps** action pattern.

![Vulnerability detail drawer — Summary, Affected component, References section restored, and Next steps action pattern](/img/screenshots/release-v2-4-0-drawer-vuln.png)

### Filter, columns, palette (W9-#52 / #54)

Three additions land on every list tab (Vulnerabilities, Components, Projects, Scans):

- **+ Add filter** chip — discoverable, multi-field, persistent in the URL.
- **Columns picker** — show / hide / reorder list columns; saved per user.
- **⌘K command palette** — search projects, scans, settings, and admin areas from any page.

### Maven classifier-aware scanning (W8-#46)

Maven artefacts with a `classifier` (`-sources`, `-javadoc`, `-tests`, custom shadow JARs) now scan correctly. Earlier builds collapsed classifier variants into the base coordinate; v2.4.0 preserves the classifier in the `purl` and matches CVEs per variant.

## Breaking changes

### Dependency-Track removed

DT containers, the `/admin/dt` page, the `dt_*.py` Celery tasks, the `integrations/dt/` Python module, and the `docker-compose.dt.yml` overlay are all deleted. CVE matching now flows through `apps/backend/integrations/trivy.py` → `trivy sbom` → `vulnerability_findings`.

What this means for you:

- **Existing data preserved.** `vulnerability_findings` rows from v2.3 stay in place. After upgrade the automatic re-match beat repopulates them from Trivy on the next refresh — typically within hours.
- **Frontend `/admin/dt` is gone.** Operators land on **/admin/health** instead. The forthcoming **Vulnerability data** card (W6-#43e follow-up) will surface the Trivy DB freshness here.
- **Audit log preserved.** Historical `dt_*` action rows are intact as historical fact; the admin filter UI marks them deprecated.

### Environment variables

| Removed (DT) | Replaced by (Trivy) |
|---|---|
| `DT_URL` | (none — Trivy DB is local to the worker) |
| `DT_API_KEY` | (none) |
| `DT_REQUEST_TIMEOUT_SECONDS` | `TRIVY_TIMEOUT_SECONDS` (default 300) |
| `DT_BREAKER_FAILURE_THRESHOLD` | (none — no remote engine) |
| `DT_BREAKER_COOLDOWN_SECONDS` | (none) |
| `DT_HEALTH_ENDPOINT` | (none) |
| `DT_AUTO_RESTART` | (none) |
| `DT_ORPHAN_AUTODELETE` | (none — no orphan model) |
| — | `TRIVY_DB_REPOSITORY` (default `ghcr.io/aquasecurity/trivy-db`) |
| — | `TRIVY_DB_REFRESH_HOURS` (default `168`) |
| — | `TRIVY_DB_CACHE_DIR` (default `/var/lib/trivy`) |

`bash scripts/upgrade.sh` comments out the `DT_*` keys in your existing `.env` automatically and the worker reads the `TRIVY_*` defaults from `docker-compose.yml` — no manual edit is needed when you use the script. If you upgrade by hand (custom CI / Helm), see the "Manual `.env` migration" callout below.

See [Environment variables — Vulnerability data](../reference/env-variables.md#vulnerability-data) for the canonical reference.

### Helm chart 0.3.0

The chart bumps to **0.3.0** with `appVersion: "2.4.0"`. Concretely:

- **Removed:** `env.dt.url`, `env.dt.apiKey`, the `DT_API_KEY` entry in the chart Secret, and the `DT_URL` ConfigMap entry. (There was never a `dtrack-api` Deployment in the chart — DT was always external — so no workload object is deleted.)
- **Added:** `env.trivy.dbRepository` (default `ghcr.io/aquasecurity/trivy-db`), `env.trivy.dbRefreshHours` (`168`), `env.trivy.dbCacheDir` (`/var/lib/trivy`), `env.trivy.timeoutSeconds` (`300`).
- **Bumped:** `image.tag` default `2.0.0` → `2.4.0`.

Upgrade in place:

```bash
helm upgrade trustedoss oci://ghcr.io/trustedoss/charts/trustedoss \
  --version 0.3.0 \
  -n trustedoss \
  -f my-values.yaml
```

If your existing `my-values.yaml` carries `env.dt.*` keys, drop them — Helm 0.3.0 does not define them and the upgrade would fail strict-mode validation. See [Install on Kubernetes with Helm](../installation/helm.md).

### CI integration — `scan-kind` text only

The `cdxgen + scancode + DT` description on the GitHub Action `scan-kind=source` input now reads `cdxgen + scancode + Trivy`. The contract is unchanged — existing workflows keep working without edits.

## Migration from v2.3.x

The shape of the upgrade is **single-direction with no manual data migration**. The migrating worker re-pulls the Trivy DB on first boot, the beat task re-matches every project's most-recent SBOM, and the new findings stream in over the following hours.

High-level (one-time, run on the host):

1. **Back up.** Take a fresh backup before upgrading — `/admin/backup → Run manual backup now`, or `bash scripts/backup.sh`. Verify the `manifest.json` lands on off-host storage.
2. **Pull the v2.4.0 images.** `git pull && git checkout v2.4.0`, then `docker-compose pull` (or set `IMAGE_TAG=2.4.0` in `.env` for the no-clone install path).
3. **Run `bash scripts/upgrade.sh`.** The script detects a v2.3 deployment automatically (DT keys in `.env` and/or a `dtrack-api` container present) and runs a 5-step DT removal prelude before pulling the new images:

   | Step | Action |
   |---|---|
   | 2.1 | Polls `celery inspect active` for up to **10 minutes** waiting for in-flight scans to drain. Set `UPGRADE_SKIP_DRAIN=1` to skip the wait; the script falls back to a prompt when the queue is still busy after the timeout. |
   | 2.2 | `docker rm -f dtrack-api` — the container is stopped and removed; **the docker volume is preserved** so step 2.3 can archive it. |
   | 2.3 | Prompts to archive the detected DT volume (`*dt-data*`) to `./backup/dt-volume-<timestamp>.tar.gz`. **Default: N** (no archive, volume preserved as-is). The volume is never deleted automatically. |
   | 2.4 | Idempotently comments out every `DT_*` line in `.env` and `.env.example` with a `# removed in v2.4.0 (W6-#43d)` marker. |
   | 2.5 | Prints a reminder pointing operators to **/admin/health → Trigger full re-match** (W6-#43e) for an immediate full re-match. Otherwise the 6-hourly re-match beat catches up automatically. |

   After the prelude, the script purges any straggling `trustedoss.dt_*` task names from the broker (NACK-loop guard), brings up v2.4.0, runs `alembic upgrade head`, and waits for `/health` to return 200.

4. **Verify.** Open `/admin/health` — every row green. Open any project — Vulnerabilities tab populates within minutes of the first re-match cycle.
5. **(Optional) Drop the DT volume.** Once you have archived (step 2.3) and confirmed nothing is missing, `docker volume rm trustedoss_dt-data` (the actual name varies — check `docker volume ls | grep dt`).

:::info Manual `.env` migration (custom CI / Helm — no `upgrade.sh`)
The 5-step prelude above is also the script-free runbook. If you run a custom CI / Helm-only deployment without `scripts/upgrade.sh`, do steps 2.1 → 2.4 by hand:

```bash
# 2.1 — drain (best effort; or skip via UPGRADE_SKIP_DRAIN=1)
docker-compose -f docker-compose.yml exec -T worker \
  celery -A tasks.celery_app inspect active

# 2.2 — stop & remove the dtrack-api container (volume preserved)
docker rm -f dtrack-api

# 2.3 — archive the DT volume (optional)
docker run --rm -v trustedoss_dt-data:/data:ro -v "$PWD/backup":/backup \
  busybox:1.36 tar -C /data -czf "/backup/dt-volume-$(date +%Y%m%d).tar.gz" .

# 2.4 — comment DT_* keys
sed -i.bak -E 's/^(DT_[A-Z_]+=.*)$/# \1   # removed in v2.4.0/' .env

docker-compose -f docker-compose.yml up -d
```

Then verify with `docker-compose logs --tail=200 worker | grep -i trivy` — the first DB download lands within 1–3 minutes.
:::

### Rolling back to v2.3.x

v2.4.0 does not include a v2.3 reverse migration. If you need to roll back:

1. Restore the v2.3 backup taken in step 1.
2. `git checkout v2.3.1 && docker-compose pull && docker-compose -f docker-compose.yml -f docker-compose.dt.yml up -d`.
3. Wait for DT to come up (5–10 minutes); the next hourly DT resync rebuilds the cache.

The `v2.3.1` release line carries critical-severity security fixes for 6 months after v2.4.0 GA per the SECURITY.md backport policy.
